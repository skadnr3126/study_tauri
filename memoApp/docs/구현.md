# 구현·저장 모델

이 문서는 메모앱의 자료구조 결정과 2차 이후의 로컬 저장 목표를 정리한다.

## Current Implementation Stage

현재 1차 목표는 `plan.md`에 정의한 **Flow 기본 구조 검증**이다. 이 단계에서는 메모리 안에서 Flow와 Branch를 조작하고 Outline으로 검증한다.

- 구현: Flow 생성, Flow 안에서의 Block 추가·이동·분기·삭제, tree 불변 조건 검사
- 보류: Markdown/JSON 파일 저장, autosave, Inbox, Link, Canvas, AI 기능
- 저장 모델: 1차 구조 검증이 통과한 뒤 2차에서 구현

아래의 폴더 구조와 저장 전략은 현재 구현물이 아니라 2차 목표 아키텍처다.

## Goals

- 서버 없이 사용자 컴퓨터 안에 저장한다.
- DB를 원본 저장소로 쓰지 않는다.
- Block 본문은 AI와 사람이 읽기 쉬운 Markdown으로 둔다.
- Flow는 한 방향 리스트가 아니라 가지가 뻗는 tree 구조로 표현한다.
- 완전 자유 그래프는 MVP에서 피하고, tree + auxiliary links로 보완한다.
- 앱 실행 중에는 파일을 직접 계속 만지지 않고 memory state를 조작한 뒤 저장한다.

## Core Decision: Block Is the Tree Item

`Block`은 생각의 내용이면서 동시에 Flow tree에 놓이는 유일한 카드 단위다. 과거의 `Node` 개념은 별도 엔티티로 두지 않는다.

- Block ID가 tree item ID이자 layout의 기준 ID다.
- 활성 Block은 정확히 하나의 Flow tree에 정확히 한 번 존재한다.
- Block을 생성하면 Flow 또는 Inbox에 즉시 배치한다.
- Block을 이동해도 같은 Block ID를 유지한다.
- Block을 삭제하거나 Flow를 삭제하면 해당 Block도 함께 Trash로 이동한다.
- Block을 복제할 때만 새 ID를 만든다.

Block 본문 파일과 Flow 구조 파일은 저장 형식상 분리될 수 있다. 하지만 이는 하나의 Block을 저장하기 위한 표현 분리일 뿐, 내용 원본과 배치 원본의 생명주기를 분리한다는 뜻은 아니다. Block의 생성·복제·삭제·복원은 관련 Flow 구조와 함께 하나의 작업으로 처리한다.

## Folder Layout

```text
.memo/
  blocks/
    2026/
      07/
        blk_01J2ABC.md

  flows/
    flow_inbox.json
    flow_storage_design.json

  exports/
    flow_storage_design.md

  trash/
    blocks/
    flows/

  index.json
  layout.json
```

역할:

- `blocks/*.md`: Block의 본문과 메타데이터.
- `flows/*.json`: Block ID의 순서, 분기, Flow 메타데이터, Link를 저장.
- `exports/*.md`: AI와 사람이 읽기 좋게 펼친 문서. 자동 생성물이다.
- `index.json`: Block ID와 파일 경로, 제목, 태그, 현재 Flow를 담는 재생성 가능한 캐시.
- `layout.json`: Block 위치, 접힘 상태, 선택 상태 등 UI 상태.
- `trash/`: Block과 Flow의 삭제 복구용 보관소.

## Core Concepts

### Block

`Block`은 사용자가 하나의 카드처럼 편집하고 이동할 수 있는 최소 의미 단위이자 Flow tree의 항목이다.

```md
---
id: blk_01J2ABC
type: thought
title: 파일 기반 메모 앱
created: 2026-07-11T10:30:00+09:00
updated: 2026-07-11T10:32:00+09:00
tags: [storage, memo-app]
---

DB 없이 파일 기반으로 메모 앱을 만들고 싶다.
```

Block에는 다음 정보를 저장한다.

- `id`, `type`, `created`, `updated`, `tags`, 선택적 `title`
- Markdown body와 본문 안의 `[[wikilink]]`

`title`은 목록과 Outline에서 쓰는 선택적 메타데이터다. 제목이 없으면 Markdown 본문의 첫 의미 있는 줄을 화면용 제목으로 사용하며, 이 파생 제목을 원본에 자동 저장하지 않는다.

Flow JSON은 Block ID를 사용해 순서와 branch를 표현한다. Block의 위치는 별도의 Node ID가 아니라 그 Block ID가 Flow tree에 나타난 위치로 식별한다. Block은 하나의 tree item만 가질 수 있으므로 재사용하거나 독립적으로 detach할 수 없다.

### Flow

`Flow`는 Block들의 tree 구조와 그 흐름의 메타데이터를 가진다.

```json
{
  "version": 2,
  "id": "flow_storage_design",
  "title": "메모앱 저장 구조",
  "created": "2026-07-11T10:40:00+09:00",
  "updated": "2026-07-11T10:45:00+09:00",
  "root": {
    "id": "root",
    "items": ["blk_01J2ABC", "blk_01J2GHI"],
    "branches": {
      "blk_01J2ABC": [
        {
          "id": "branch_01J2BBB",
          "title": "파일 수 문제",
          "items": ["blk_01J2DEF"],
          "branches": {}
        }
      ]
    }
  },
  "links": [
    {
      "id": "link_01J2EEE",
      "fromBlockId": "blk_01J2DEF",
      "toBlockId": "blk_01J2GHI",
      "type": "reference"
    }
  ]
}
```

의미:

- `items`: 같은 흐름에서 이어지는 Block ID의 순서.
- `branches[blockId]`: 해당 Block에서 옆으로 뻗은 branch들.
- `branch.items`: branch 안에서 이어지는 Block ID의 순서.
- `links`: tree로 표현하지 않는 보조 관계.

`flow_inbox`는 Inbox를 위한 특수 Flow다. 빠른 캡처는 이 Flow에 Block을 생성하고, Inbox에서 다른 Flow로 이동하는 것은 같은 Block의 tree 위치를 바꾸는 작업이다. 따라서 활성 상태의 고아 Block은 없다.

### Schema Migration

이전 schema가 `FlowNode { id, blockId, branches }`를 사용했다면, 새 schema에서는 Node를 만들지 않고 `blockId`를 tree item으로 사용한다.

```text
v1 FlowNode.id       -> 제거
v1 FlowNode.blockId  -> v2 items의 Block ID
v1 node.branches     -> v2 branches[blockId]
v1 layout[nodeId]    -> v2 layout[blockId]
```

마이그레이션 전에 workspace를 백업한다. 한 Block이 여러 Node에 있던 손상 또는 구 schema 데이터는 자동 선택하지 않고, 사용자에게 하나의 위치를 선택하거나 Block을 복제하도록 안내한다.

### Link

Link는 Block 사이의 느슨한 관계를 나타낸다. 생각의 전개는 tree에, 연관·합류·참조는 Link에 둔다.

```text
reference = 관련 있음
merge = 이 흐름이 저 Block으로 합류함
supports = 근거
contradicts = 반박
example = 예시
```

MVP에서는 `reference`, `merge` 정도만 둔다. Link는 Block의 생명주기를 공유하지 않는다. Block이 Trash로 이동하면 그 Block을 가리키는 Link는 함께 Trash에 보관하거나 복원 시 되살릴 수 있도록 삭제 작업에 포함한다.

## Runtime Memory State

1차에서는 파일을 읽지 않고, in-memory `WorkspaceState`를 조작한다. Flow는 중첩 객체 대신 `Branch Map + Block ID 배열`로 보관한다. 이 방식이면 Block과 Branch의 이동·삭제·검증을 직접 수행하기 쉽다.

```ts
type WorkspaceState = {
  blocks: Map<BlockId, Block>;
  flows: Map<FlowId, Flow>;
  activeFlowId?: FlowId;
  selectedBlockId?: BlockId;
};

type Block = {
  id: BlockId;
  title?: string;
  markdown: string;
  createdAt: string;
  updatedAt: string;
};

type Flow = {
  id: FlowId;
  title: string;
  rootBranchId: BranchId;
  branches: Map<BranchId, Branch>;
};

type Branch = {
  id: BranchId;
  title?: string;
  parentBranchId?: BranchId;
  parentBlockId?: BlockId;
  itemIds: BlockId[];
  childBranchIdsByBlockId: Map<BlockId, BranchId[]>;
};
```

`childBranchIdsByBlockId`는 한 Block 아래의 Branch 순서를 보존한다. root가 아닌 Branch는 어떤 Branch의 어떤 Block에서 시작됐는지 `parentBranchId`, `parentBlockId`로 표현한다.

빠른 조작을 위해 필요하면 Block ID 기준 위치 인덱스를 파생한다.

```ts
type BlockLocation = {
  flowId: FlowId;
  branchId: BranchId;
  index: number;
};
```

2차에서는 이 상태를 Markdown/JSON 파일에서 만들고, dirty state·layout·Link·저장용 index를 추가한다. 저장 구조와 메모리 구조는 반드시 같을 필요가 없다. 저장은 읽기 쉬운 tree JSON으로 두고, 메모리에서는 빠른 탐색용 index를 추가로 만든다.

## Save Strategy

변경된 것만 저장한다.

```text
Block 내용 수정 -> 해당 block.md 저장
Block 이동·분기·순서 변경 -> 해당 flow.json 저장
Block 생성·복제·삭제·복원 -> block.md와 관련 flow.json을 하나의 작업으로 저장
Block 위치의 UI 변경 -> layout.json 저장
Flow 변경 -> 해당 export.md 재생성
검색 정보 변경 -> index.json 부분 갱신 또는 재생성
```

저장 타이밍:

- 글자 입력: debounce autosave.
- Block 이동: 짧은 debounce 또는 즉시 저장.
- Flow 분리·병합·삭제: 즉시 저장 권장.
- 앱 종료 전: dirty state 전부 저장.

저장은 atomic write를 기본으로 한다. 여러 파일을 함께 바꾸는 Block 생명주기 작업은 transaction marker 또는 operation log로 재실행·복구 가능해야 한다.

```text
새 파일 또는 .tmp 파일에 먼저 쓰기
관련 Flow와 Block 파일이 준비되면 rename
성공 후 layout, index, export 갱신
```

## Export

`exports/*.md`는 자동 생성물이다. 원본이 아니다.

```md
<!-- AUTO-GENERATED. DO NOT EDIT. Source: flows/flow_storage_design.json -->

# 메모앱 저장 구조

## blk_01J2ABC

DB 없이 파일 기반으로 메모 앱을 만들고 싶다.

### Branch: 파일 수 문제

## blk_01J2DEF

block 파일이 너무 많아지면 폴더를 나눠야 한다.

Related:
- references [[blk_01J2GHI]]
```

사용자가 직접 편집하지 않으며, 깨지거나 삭제되어도 재생성 가능해야 한다.

## Integrity Checks

앱 시작 시 또는 저장 후 정기적으로 검사한다.

- 모든 활성 Block이 정확히 하나의 Flow tree 위치를 갖는가?
- 모든 Flow의 Block ID가 실제 Block 파일을 가리키는가?
- 같은 Block ID가 여러 위치에 등장하지 않는가?
- branch가 존재하는 Block ID를 가리키는가?
- Link target Block이 존재하는가?
- layout이 삭제된 Block을 가리키는가?
- Flow·Block schema version이 지원 가능한가?
- Block ID와 Flow ID가 각각 중복되지 않는가?

## Deletion Policy

Block과 tree item은 같은 생명주기다.

```text
Block 삭제 = Flow tree에서 제거 + Block 파일과 관련 Link를 Trash로 이동
Branch 삭제 = 포함된 모든 Block과 branch를 subtree 단위로 Trash로 이동
Flow 삭제 = Flow와 내부의 모든 Block을 Trash로 이동
```

바로 영구 삭제하지 않고 `trash/`로 보낸다. 삭제 전에는 branch를 다른 위치로 이동하거나 새 Flow로 분리할 수 있다. 중간 Block 하나를 지울 때 자식 branch를 유지할 필요가 있으면, 먼저 branch를 명시적으로 이동해야 한다. MVP에서는 삭제 확인과 undo를 강하게 둔다.

## Performance Notes

파일이 많아지는 것 자체는 괜찮지만 한 폴더에 너무 많이 몰리면 문제가 된다.

```text
blocks/
  2026/
    07/
      blk_01J2ABC.md
```

Block ID와 파일 경로는 분리하고 `index.json`이 ID → path를 담당한다. 화면은 부분 렌더링할 수 있으며, MVP에서는 Flow 단위 전체 레이아웃 계산으로 시작한다. export는 캐시이므로 관련 Flow 전체를 재생성한다.

## Current Decision Summary

```text
Source of truth:
  blocks/*.md              # Block 본문과 메타데이터
  flows/*.json             # 같은 Block의 tree 위치와 관계

Generated:
  exports/*.md
  index.json

UI state:
  layout.json              # Block ID 기준

Lifecycle:
  Block = Flow tree item
  활성 Block은 한 Flow에 한 번만 존재
  Inbox는 특수 Flow

Runtime:
  Map<BlockId, Block>
  nested Flow tree of Block IDs
  runtime indexes keyed by Block ID

Main structure:
  tree

Auxiliary relation:
  links

  not required for MVP
```
