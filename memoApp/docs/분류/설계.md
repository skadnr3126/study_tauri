# Flow 기반 메모앱 현재 설계 

## 현재 구현 단계

현재 구현 기준은 `plan.md`의 1차 목표다. 우선 in-memory Flow tree와 Outline 조작을 검증한다.

```text
구현: Flow 생성, Block/Branch 생성·이동·삭제, tree 검증
보류: 파일 저장, Inbox, Link, Canvas, AI, 성능 최적화
```

이 문서의 저장 구조, Inbox, AI 관련 내용은 1차 검증 후 적용할 장기 설계다.

## 1. 프로젝트 개요

이 프로젝트는 사용자의 생각, 질문, 학습 과정, AI와의 대화를 낮은 비용으로 기록하고, 나중에 해당 사고 흐름으로 쉽게 다시 진입할 수 있도록 돕는 로컬 기반 메모앱을 만드는 프로젝트다.

이 앱은 완성된 문서를 작성하거나 지식을 깔끔하게 분류하는 것을 가장 중요한 목표로 삼지 않는다.

핵심 목표는 다음과 같다.

> **빠르게 지나가는 생각의 원본과 그 생각이 이어진 흐름을 보존하고, 사용자가 나중에 다시 생각을 이어갈 수 있도록 돕는다.**

앱은 사용자를 대신해 생각하는 도구가 아니다.

사용자가 생각에 더 집중할 수 있도록 기록 비용, 기억 복구 비용, 구조 변경 비용을 낮추는 보조 도구다.

---

# 2. 앱의 핵심 철학

이 앱의 핵심 철학은 **흐름 기록**이다.

여기서 흐름은 메모가 단순히 시간순으로 쌓이는 것을 의미하지 않는다.

흐름은 다음과 같은 사고의 연결을 의미한다.

* 어떤 관찰에서 생각이 시작되었는가
* 한 생각이 어떤 질문을 만들었는가
* 질문이 다음 생각이나 대화를 어떻게 만들었는가
* 어떤 근거와 반론을 거쳐 생각이 확장되었는가
* 어떤 지점에서 새로운 가지가 생겼는가
* 사용자가 어디까지 생각했고 무엇을 이어가려 했는가

이 앱은 완성된 결과물보다 **사고의 원천과 전개 과정**을 중요하게 본다.

따라서 다음 원칙을 따른다.

* 생각의 주체는 사람이다.
* 앱은 생각을 돕는 보조 도구다.
* AI는 사고의 주체가 아니라 재료 제공자다.
* 원본 기록과 원본 Flow는 최대한 보존한다.
* 요약과 정리된 문서는 원본이 아니라 파생물이다.
* 정리보다 캡처와 보존이 먼저다.
* 구조를 입력 순간부터 강제하지 않는다.
* 사용자는 생각에 집중하고 앱은 흐름을 보존한다.
* 먼저 붙잡고, 나중에 구조화한다.
* Capture first, structure later.

# 3. 인간 성장과 학습 관점

이 앱의 장기적인 목적은 단순한 정보 저장이 아니라 **인간의 성장 루프를 증폭하는 것**이다.

인간의 성장은 하나의 순환 과정으로 볼 수 있다.

```text
경험
→ 주의
→ 이해
→ 생각
→ 학습
→ 기억
→ 직관
→ 행동
→ 피드백
→ 새로운 경험
```

이 과정에서 여러 인간 능력이 성장 루프를 증폭하는 레버로 작동한다.

* Attention은 중요한 입력을 선택하게 한다.
* Working Memory는 여러 정보를 동시에 다루며 더 깊게 생각하도록 돕는다.
* Long-term Memory는 배운 것과 생각한 것을 축적한다.
* Reflection은 경험과 피드백을 다시 해석하게 한다.
* Motivation은 성장 루프를 계속 돌게 한다.

노트는 이 능력들을 직접 향상시키는 약이나 훈련 도구는 아니다.

하지만 생각을 외부화하고, 다시 보고, 이동하고, 연결하고, 수정할 수 있게 함으로써 이러한 능력이 더 잘 작동할 수 있는 환경을 만든다.

특히 이 앱은 다음 비용을 줄이는 것을 중요하게 본다.

* 떠오른 생각을 기록하는 비용
* 과거 생각을 찾는 비용
* 중단된 생각의 맥락을 복구하는 비용
* 복잡한 생각을 머릿속에 유지하는 비용
* 생각을 재배치하고 수정하는 비용
* AI 대화에서 중요한 흐름을 다시 찾는 비용

# 4. 노트에 대한 관점

기존 노트앱은 주로 정보를 저장하고 정리하는 도구로 설계되어 있다.

이 앱은 노트를 다음과 같이 정의한다.

> **노트는 인간의 생각을 외부화하고, 흐름을 보존하며, 다시 생각할 수 있게 만드는 인터페이스다.**

사람은 페이지나 폴더 단위로만 생각하지 않는다.

생각은 이어지고, 갈라지고, 다른 생각과 연결되며, 때로는 다시 합쳐진다.

따라서 이 앱은 완성된 문서보다 다음 요소를 중요하게 본다.

* 생각의 최소 의미 단위
* 생각이 이어진 순서
* 중간에서 발생한 분기
* 느슨한 연관 관계
* 수정과 이동의 편리함
* 원래 흐름으로 다시 돌아갈 수 있는 구조

다만 사용자가 생각을 기록할 때마다 유형, 상태, 확신도, 변화 이유 등을 입력하도록 강제하지 않는다.

사용자에게 보이는 구조는 단순해야 한다.

```text
Block
Flow
Branch
Link
Inbox
```

생성 시각, 수정 시각, 이동 기록, 참조 관계, 검색 인덱스 같은 정보는 앱이 내부적으로 자동 관리한다.

핵심 원칙은 다음과 같다.

> **구조를 입력하게 하지 말고, 사용자의 자연스러운 기록과 조작에서 구조가 남게 한다.**

# 5. 핵심 개념

## 5.1 Block

Block은 사용자가 하나의 카드처럼 이동하고 연결할 수 있는 최소 의미 단위다.

한 문장 단위로 강제하지 않는다.

하나의 Block은 일반적으로 다음 중 하나를 담을 수 있다.

* 생각
* 질문
* 관찰
* 주장
* 근거
* 반론
* 학습 내용
* AI 질문과 답변
* 임시 메모

Block은 내용의 원본이면서 Flow tree에 놓이는 유일한 카드 단위다. 별도의 Node 객체나 Node ID는 두지 않는다.

Block 자체에는 다음 정보를 저장할 수 있다.

* 고유 ID
* 제목
* 본문
* 생성 시각
* 수정 시각
* 태그
* 타입
* 내부 링크
* 첨부파일 참조

`title`은 선택적 메타데이터다. 제목을 입력하지 않으면 본문의 첫 의미 있는 줄을 화면용 제목으로 사용한다. 이 표시 제목은 파생값이며 원본에 자동 저장하지 않는다.

1차 구현에서는 `id`, 선택적 `title`, `markdown`, 생성·수정 시각만 사용한다. 태그·타입·출처·첨부파일은 저장과 검색 기능을 추가하는 후속 단계에서 확장한다.

순서와 branch 관계는 Block ID를 항목으로 갖는 Flow tree에 저장한다. 이는 Block과 별개로 수명 주기를 갖는 Node가 아니라, 같은 Block의 구조상 위치다.

다음 UI 정보는 Block 원본에 두지 않는 것이 좋다.

* 캔버스상의 위치
* 접힘 상태
* 선택 상태

위 정보는 Block의 의미가 아니라 화면에서 어떻게 보이는지에 관한 정보이기 때문이다. 반면 Block의 생성·복제·삭제·복원은 Flow tree 위치와 항상 함께 처리한다. 활성 Block은 정확히 하나의 Flow에 한 번만 존재한다.

---

## 5.2 Flow

Flow는 Block들이 하나의 사고 흐름 안에서 어떻게 이어지고 분기되는지를 나타내는 구조다.

Flow는 단순 폴더가 아니다.

Flow는 다음을 표현한다.

* 생각이 이어지는 주 흐름
* 특정 생각에서 파생된 가지
* 가지의 분리와 확장
* Block의 순서
* 사용자가 현재 구성한 사고 구조

Flow는 Block ID의 순서와 branch를 저장한다. 따라서 Block은 이동할 때 ID를 유지하고 위치만 바뀌며, Block을 삭제하면 Flow에서도 함께 사라진다.

기본 구조는 완전 자유 그래프보다 **트리와 브랜치 구조**에 가깝다.

```text
생각 A
→ 생각 B
→ 생각 C
   └→ 질문 D
      → 생각 E
```

트리를 기본 구조로 두는 이유는 다음과 같다.

* 사고 흐름을 이해하기 쉽다.
* 키보드 조작과 렌더링이 단순하다.
* 가지를 접거나 분리하기 쉽다.
* AI용 문서로 펼치기 쉽다.
* 사용자가 구조를 예측하기 쉽다.

트리로 표현하기 어려운 느슨한 관계나 참고 관계는 별도의 Link로 처리한다.

```text
핵심 전개 구조 = Tree
부가적인 연관 관계 = Link
```

---

## 5.3 Link

Link는 Block 사이의 느슨한 관계를 나타낸다.

예를 들면 다음과 같다.

* 관련 있음
* 참고
* 반대되는 생각
* 비슷한 주제
* 나중에 연결할 생각
* 다른 Flow와의 연관

Flow의 트리는 생각의 전개 구조이고, Link는 의미적 연관 관계다.

둘을 하나의 자유 그래프로 섞으면 구조를 이해하기 어려워질 수 있으므로 분리한다.

---

## 5.4 Inbox

모든 기록을 처음부터 Flow로 만들도록 강제하면 캡처 비용이 증가한다.

따라서 Inbox가 필요하다.

Inbox는 Flow에 아직 분류하지 않은 Block을 보여 주는 **특수 Flow**다. 일반 Flow와 같은 tree 모델을 사용하므로 활성 Block은 Inbox를 포함해 반드시 하나의 Flow에 속한다.

Inbox에는 다음이 저장된다.

* 어떤 Flow에 넣어야 할지 아직 모르는 생각
* 흐름 없이 떠오른 질문
* 단편 메모
* 나중에 판단할 기록
* 빠른 캡처
* 아직 구조화되지 않은 아이디어

기본 동작은 다음과 같다.

```text
생각이 떠오름
→ 전역 단축키
→ 빠른 입력
→ Inbox Flow에 Block 생성
→ 나중에 Flow로 이동 또는 연결
```

Flow 안에서 기록한 생각은 현재 Flow에 바로 추가할 수 있다.

따라서 저장 위치 원칙은 다음과 같다.

* 일반 빠른 캡처: Inbox
* Flow 편집 중 기록: 현재 Flow
* AI 대화 자동 수집: AI Conversation Flow

# 9. 저장 구조

## 9.1 저장 원칙

현재 저장 방향은 SQLite를 원본 저장소로 사용하지 않고, 사용자가 직접 읽을 수 있는 파일을 원본으로 두는 것이다.

핵심 원칙은 다음과 같다.

* 사용자의 생각 원본은 Markdown으로 유지한다.
* Flow 구조는 JSON으로 유지한다.
* 앱 실행 중에는 메모리 상태에서 조작한다.
* 검색과 화면 렌더링을 위한 데이터는 캐시로 취급한다.
* AI용 조립 문서는 원본이 아니라 자동 생성물이다.
* 원본과 파생 데이터를 명확하게 구분한다.

초기 설계에 포함되어 있던 SQLite는 현재 방향과 충돌하므로 정리할 필요가 있다.

SQLite를 완전히 사용하지 않거나, 사용하더라도 검색 인덱스와 성능 최적화를 위한 재생성 가능한 캐시로만 사용한다.

SQLite가 파일 원본과 별도의 진실 공급원이 되어서는 안 된다.

## 9.2 역할 분리

```text
blocks/*.md
= 생각의 원본

flows/*.json
= Block들의 순서, 가지, 구조

exports/*.md
= AI와 사람이 읽기 좋은 펼친 문서

index.json 또는 캐시 DB
= 검색과 목록 표시를 위한 재생성 가능한 인덱스

layout.json
= 위치, 접힘, 확대 상태 등 UI 표현

memory state
= 앱 실행 중 실제 편집과 조작에 사용하는 상태

operations
= Undo/Redo와 실패 복구에 필요한 조작 기록

trash
= 안전한 삭제
```

## 9.3 추천 디렉토리

```text
.memo/
├─ blocks/
│  └─ 2026/
│     └─ 07/
│        ├─ blk_01J2ABC.md
│        └─ blk_01J2DEF.md
│
├─ flows/
│  ├─ flow_inbox.json
│  ├─ flow_storage_design.json
│  └─ flow_product_idea.json
│
├─ exports/
│  ├─ human/
│  └─ ai/
│
├─ operations/
│
├─ trash/
│  ├─ blocks/
│  └─ flows/
│
├─ sources/
│  └─ codex-cursors.json
│
├─ index.json
├─ layout.json
├─ schema.json
└─ workspace.json
```

Block 파일이 많아질 수 있으므로 한 폴더에 모두 넣지 않고 연도와 월 등으로 분산한다.

Block ID와 파일 경로는 분리한다.

```text
blockId = blk_01J2ABC
path = blocks/2026/07/blk_01J2ABC.md
```

Flow는 경로가 아니라 Block ID만 항목으로 참조한다. 같은 Block ID는 활성 Flow tree 전체에서 한 번만 나타난다.

파일 위치가 바뀌어도 Flow 구조가 깨지지 않도록 한다.

## 9.4 앱 실행 중 상태

앱 실행 시 파일을 읽어 메모리에 기준 상태를 만든다.

```text
Disk Files
→ Load
→ Canonical Memory State
→ UI State와 Render Tree 생성
```

사용자가 편집할 때마다 파일을 직접 조작하는 것이 아니라, 먼저 메모리 상태를 변경한다.

```text
사용자 조작
→ Memory State 변경
→ Dirty 표시
→ 자동 저장
→ 필요 시 Export와 Index 재생성
```

저장 방식은 다음과 같다.

* Block 내용 수정: 해당 Block 파일만 저장
* Block 이동·분기·순서 변경: 해당 Flow 파일만 저장
* Block 생성·복제·삭제·복원: Block 파일과 관련 Flow 파일을 하나의 복구 가능한 작업으로 저장
* UI 위치 변경: Layout 저장
* Flow 분리 및 병합: 관련 파일들을 하나의 작업으로 저장
* Export: 원본 저장 후 재생성
* Index: 변경 후 갱신하거나 필요 시 전체 재생성

여러 파일이 동시에 변경되는 작업에는 atomic write와 최소한의 트랜잭션 복구가 필요하다.

## 12.2 Flow 화면

Flow 화면은 두 가지 표현 방식을 가질 수 있다.

### Outline View

세로 흐름과 분기를 빠르게 읽는 방식이다.

특히 AI 대화, 학습 과정, 긴 텍스트에는 Outline이 적합하다.

### Canvas View

Block을 공간적으로 이동하고 가지를 시각적으로 확인하는 방식이다.

브레인스토밍, 설계, 자유로운 사고 구조에는 Canvas가 적합하다.

두 화면은 서로 다른 데이터를 가지는 것이 아니라 같은 Flow를 다르게 표현하는 View여야 한다.

```text
같은 Flow
→ Outline View
→ Canvas View
→ Export View
```

AI 대화 Flow는 MVP에서 Outline View를 우선하고, 수동 Flow는 단순 Outline과 제한된 Canvas 중 하나로 시작하는 것이 현실적이다.
