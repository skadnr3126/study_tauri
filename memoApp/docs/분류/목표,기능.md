# 14. MVP 범위

## 현재 1차 구현 범위

현재 구현은 아래의 장기 MVP 전체가 아니라 `plan.md`의 Flow 구조 검증만을 대상으로 한다.

- in-memory Flow 생성과 전환
- 선택적 제목과 Markdown 본문을 가진 Block 생성·편집
- Block 순서 변경, Branch 생성·이동, 하위 Branch를 포함한 삭제
- Outline UI와 tree 불변 조건 검사

Markdown/JSON 저장, Inbox, Quick Capture, Codex 수집, Export, Undo/Trash, Canvas는 모두 후속 단계다.

## 장기 MVP 범위

장기 MVP의 목적은 기능을 많이 넣는 것이 아니라 다음 가설을 검증하는 것이다.

> **사용자가 별도로 정리하지 않아도 생각과 AI 대화가 낮은 비용으로 쌓이고, 나중에 Flow를 통해 다시 보는 것이 실제로 사고 재진입 비용을 낮추는가?**

이를 위해 MVP에는 다음 기능이 필요하다.

1. Tauri 기반 데스크톱 앱
2. Markdown Block 저장
3. JSON Flow 저장
4. Inbox
5. Flow 목록
6. 단순 Outline 기반 Flow 보기
7. Block 추가, 이동, 분기
8. 전역 단축키 Quick Capture
9. 자동 저장
10. Codex JSONL Watcher
11. Codex 질문과 최종 답변을 AI Turn Block으로 생성
12. AI 답변 접기와 펼치기
13. 자동 수집 On/Off
14. Export 생성
15. 기본적인 Trash와 Undo

MVP에서 제외할 수 있는 기능은 다음과 같다.

* 고급 AI 자동 정리
* AI가 Flow를 자동 재구성하는 기능
* 음성 입력
* 이미지와 자유 스케치
* 브라우저 확장
* MCP
* 클라우드 동기화
* 협업
* 고급 의미 검색
* 완전 자유 그래프
* 자동 회고와 정교한 Resurfacing
* 복잡한 버전 관리 UI

# 6. 주요 목표

## 6.1 생각 캡처 비용 최소화

기록 비용이 높으면 생각은 사라진다.

따라서 앱은 다음을 지원해야 한다.

* 전역 단축키
* 즉시 열리는 빠른 입력창
* 입력창이 열리자마자 타이핑 가능한 상태
* 키보드 중심 조작
* 자동 저장
* 최소한의 확인 과정
* 마우스 사용 최소화
* 최근 Flow 또는 Inbox로 빠른 저장
* 추후 음성 입력 확장 가능성

이 과정에서 사용자가 태그, 폴더, Flow, 타입 등을 반드시 선택하도록 해서는 안 된다.

---

## 6.2 생각 외부화 지원

노트의 첫 번째 역할은 머릿속의 생각을 밖으로 꺼내는 것이다.

작업 기억에만 존재하던 생각이 Block으로 외부화되면 사용자는 다음 행동을 할 수 있다.

* 다시 읽기
* 비교하기
* 이동하기
* 갈라놓기
* 합치기
* 연결하기
* 수정하기
* 다른 Flow로 이동하기

앱은 텍스트 작성 기능보다 **생각 단위 조작 비용을 낮추는 것**에 집중해야 한다.

---

## 6.3 구조화 지원

구조화는 입력 전에 요구되는 절차가 아니라 기록 이후에 가능한 조작이어야 한다.

사용자는 먼저 기록하고 나중에 다음 작업을 할 수 있어야 한다.

* Inbox의 Block을 Flow로 이동
* 두 Block 사이에 새 Block 삽입
* 기존 Block에서 새 가지 생성
* 가지를 독립된 Flow로 분리
* 여러 Block의 순서 변경
* 관련 Block끼리 Link 생성
* Flow 일부를 접거나 펼치기
* Flow를 문서 형태로 펼쳐보기

구조화 기능은 강력해야 하지만 사용자의 기록을 방해해서는 안 된다.

---

## 6.4 생각 복구 비용 감소

사용자가 중단한 생각으로 돌아왔을 때 빠르게 맥락을 복구할 수 있어야 한다.

Flow를 열었을 때 다음을 알 수 있어야 한다.

* 무엇을 생각하고 있었는가
* 어떤 순서로 생각이 이어졌는가
* 어떤 질문이 다음 생각을 만들었는가
* 어디에서 가지가 갈라졌는가
* 어떤 자료나 AI 답변을 참고했는가
* 마지막으로 어디까지 진행했는가
* 이어갈 수 있는 지점은 어디인가

앱은 단순 검색 결과뿐 아니라 **흐름 자체를 복구하는 인터페이스**를 제공해야 한다.

---

## 6.5 수정과 이동 최적화

생각은 처음부터 완성되어 있지 않다.

따라서 Block을 쉽게 수정하고 이동하고 분리하고 연결할 수 있어야 한다.

중요한 조작은 다음과 같다.

* Block 내용 수정
* Block 이동
* Block 순서 변경
* Branch 생성
* Branch 분리
* Flow 병합
* Block 복제
* Link 생성
* Block과 Branch를 Trash로 이동
* Block 삭제와 영구 삭제 구분
* Undo와 Redo

Block은 내용과 Flow tree 위치가 같은 생명주기를 갖는다. 활성 Block은 Inbox를 포함해 정확히 하나의 Flow에 존재하며, Flow에서 Block만 분리해 고아 상태로 남길 수 없다.

삭제 정책은 다음처럼 구분한다.

```text
Block 삭제 = Flow tree에서 제거 + Trash로 이동
영구 삭제 = Trash에서 완전히 제거
```

다른 Flow로 옮기려면 삭제하지 않고 같은 Block을 이동한다. 빠른 캡처는 Inbox 특수 Flow에 생성한다.

# 7. AI 대화 기록

## 7.1 목적

AI 대화 기록 기능의 목적은 AI가 사용자의 대화를 자동으로 완벽하게 정리하는 것이 아니다.

목적은 AI와 대화하면서 발생한 질문, 답변, 사고 변화의 원본을 보존하고, 나중에 다시 접근하기 쉽게 만드는 것이다.

일반적인 AI 채팅 UI는 긴 세로 스크롤 구조를 가진다.

대화가 길어질수록 다음 문제가 발생한다.

* 이전 질문을 찾기 어렵다.
* 주제가 어디서 바뀌었는지 알기 어렵다.
* 중요한 답변이 묻힌다.
* 질문 간 연결이 보이지 않는다.
* 대화 중 발생한 새로운 가지를 분리하기 어렵다.

따라서 AI 대화를 Flow로 가져와 다시 볼 수 있어야 한다.

---

## 7.2 원칙

AI 대화 기록은 다음 원칙을 따른다.

* 사용자 질문 원문 보존
* AI 답변 원문 보존
* 대화 순서 보존
* 세션 정보 보존
* 원본 로그 수정 금지
* 요약이 원본을 대체하지 않음
* 복잡한 자동 구조화는 초기 단계에서 최소화
* 사용자가 나중에 직접 분리하고 연결할 수 있음

중요한 구분은 다음과 같다.

```text
원본 대화 = Source Material
Flow View = 재접근을 위한 표현
Export와 요약 = 파생물
```

---

## 7.3 AI Turn Block

MVP에서는 사용자 질문과 AI 최종 답변의 한 쌍을 하나의 AI Turn Block으로 볼 수 있다.

```text
AI Turn Block
├─ 사용자 질문
├─ AI 답변
├─ 세션 ID
├─ 메시지 ID
├─ 생성 시각
└─ 원본 로그 참조
```

AI Turn Block 제목은 사용자 질문의 첫 줄이나 일정 길이까지의 앞부분으로 자동 생성할 수 있다.

AI 답변은 기본적으로 접어두고 필요할 때 펼쳐볼 수 있도록 한다.

Tool output, commentary, 중간 추론 정보 등은 삭제하지 않고 별도 원본으로 보존하되, 기본 화면에서는 숨기거나 접어둘 수 있다.

# 8. Codex 로그 수집

MVP에서는 Codex CLI의 로컬 세션 로그를 첫 번째 수집 대상으로 삼는다.

기본 구조는 다음과 같다.

```text
Codex CLI
→ ~/.codex/sessions/**/*.jsonl
→ File Watcher
→ Importer
→ Parser
→ AI Turn Block 생성
→ AI Conversation Flow에 표시
```

구현 흐름은 다음과 같다.

1. 사용자가 Codex 로그 폴더를 연결한다.
2. 앱이 로그 폴더를 감시한다.
3. 새 JSONL 파일 생성이나 기존 파일 변경을 감지한다.
4. 이전에 읽은 위치 이후의 내용만 읽는다.
5. JSONL을 한 줄씩 파싱한다.
6. 사용자 메시지와 AI 최종 답변을 추출한다.
7. 질문과 답변을 TurnNode로 만든다.
8. 세션에 대응하는 Flow에 추가한다.

각 로그 파일에는 읽은 위치를 나타내는 Cursor가 필요하다.

```ts
type LogCursor = {
  sourceId: string;
  filePath: string;
  lastOffset: number;
  lastImportedMessageId?: string;
  updatedAt: string;
};
```

이 구조를 사용하면 앱을 다시 실행해도 이어서 읽을 수 있고, 전체 로그를 매번 다시 파싱하지 않아도 된다.

로그 포맷이 바뀔 수 있으므로 importer와 parser는 앱의 핵심 저장 모델과 분리한다.

```text
Source Adapter
→ Normalized Import Event
→ Block / AI Turn Block 생성
```

이렇게 하면 나중에 Claude, ChatGPT, Cursor 등 다른 소스를 추가하기 쉽다.

# 10. Export와 AI 접근

작은 Block 파일이 많으면 AI가 직접 전체 흐름을 읽기 어렵다.

따라서 Flow 구조에 따라 Block을 펼친 Export 문서를 자동 생성한다.

```text
Block 원본 + Flow 구조
→ Export 생성
→ 사람 또는 AI가 읽음
```

Export는 다음 원칙을 따른다.

* 언제든 재생성 가능
* 사용자가 직접 편집하지 않음
* 원본으로 취급하지 않음
* 사람용과 AI용 형식을 다르게 만들 수 있음
* Flow 전체 또는 일부 Branch만 Export 가능
* 질문과 답변 원문을 포함하는 옵션 제공
* Block ID와 출처 정보 포함 가능

파일 상단에는 자동 생성물임을 표시한다.

```html
<!-- AUTO-GENERATED. DO NOT EDIT. -->
```

장기적으로 Export는 단순 Markdown 한 종류가 아니라 다양한 Projection으로 확장할 수 있다.

* 전체 사고 흐름
* 특정 Branch
* AI 컨텍스트용 조립본
* 질문만 모은 문서
* 학습 자료
* 최종 문서 초안

# 13. 핵심 키보드 조작

키보드 중심 UX를 위해 다음 조작을 고려할 수 있다.

```text
전역 단축키
= Quick Capture 열기

Enter
= 같은 흐름의 다음 Block 추가

Tab
= 현재 Block에서 Branch 생성

Shift + Tab
= 상위 흐름으로 이동

Ctrl/Cmd + Enter
= 현재 Block 저장 또는 확정

방향키
= Block 선택 이동

Ctrl/Cmd + 방향키
= Block 순서 이동

Delete
= Block과 하위 branch를 Trash로 이동
```

정확한 단축키는 구현과 사용성 테스트를 거쳐 조정한다.

중요한 것은 구조 조작을 위해 마우스가 반드시 필요하지 않도록 하는 것이다.
