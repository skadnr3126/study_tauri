Flow Memo, 어디서부터 읽을까?

이 프로젝트의 핵심은 “생각 카드(Block)를 순서와 갈래(Branch)로 묶는 트리”입니다. UI를 먼저 따라가기보다, 그 트리를 안전하게 바꾸는 규칙부터 이해하면 이후 기능을 판단하기 쉬워집니다.

먼저 알아둘 현재 상태

앱은 React + TypeScript로 동작하며 Tauri 껍데기를 갖고 있습니다. 하지만 메모, Flow, 캔버스 위치는 모두 브라우저 메모리에만 있습니다. 새로고침하거나 앱을 종료하면 사라집니다. Rust 쪽의 greet 명령은 현재 화면에서 호출되지 않습니다.

설계 문서에는 “Canvas는 후속 단계”라고 적힌 부분이 있지만, 실제 구현에는 이미 Canvas UI가 들어가 있습니다. 코드를 읽을 때는 실제 코드가 현재 동작, 문서는 의도와 다음 단계라고 분리해 보세요.

가장 중요한 경계는 이것입니다: App.tsx는 “사용자 행동을 받는 곳”, flow.ts는 “그 행동이 구조를 망가뜨리지 않게 만드는 곳”입니다.

  2. application/src/domain/flow.ts 1–253행

  3. application/src/domain/flow.ts 255–479행
    이동·삭제·검증을 읽습니다. 여기서 “자기 하위 Branch로 이동 금지”와 “삭제는 이후 노드와 하위 가지까지 포함” 규칙을 확인합니다.
  4. application/src/App.tsx
    React 상태와 runCommand, 컴포넌트를 조립하는 방식을 봅니다. UI가 도메인 함수를 어떻게 부르는지가 보입니다.

핵심 데이터 모델

WorkspaceState
├─ blocks: Map<BlockId, Block>     # 메모 내용 원본
├─ flows: Map<FlowId, Flow>         # 생각의 트리들
├─ activeFlowId                      # 현재 열어 둔 Flow
└─ selectedBlockId                   # 현재 선택한 Block

Flow → rootBranchId + branches: Map<BranchId, Branch>
Branch → itemIds: BlockId[] + childBranchIdsByBlockId

Block에는 내용만 있고, “어느 Flow의 몇 번째인가”는 Branch의 itemIds에 있습니다. 이 분리가 가장 중요한 설계 결정입니다.

읽으며 답해 볼 질문: B 아래에 새 갈래를 만들면, 새 Block은 어디에 들어가고 B와 새 Branch의 연결 정보는 어디에 저장될까? → createBranchWithBlock에서 확인하세요.

코드에서 특히 주의할 지점

1. cloneWorkspace

명령은 원래 상태를 직접 바꾸지 않기 위해 Map·배열을 복제합니다. React가 변경을 안정적으로 감지하도록 하는 현재 프로젝트의 상태 갱신 규칙입니다.

2. 삭제의 의미

deleteBlockSubtree는 선택 Block 하나만 지우지 않습니다. 같은 Branch에서 그 뒤에 이어지는 Block과 그들의 하위 Branch까지 삭제합니다. UI 문구도 이 규칙을 명확히 해야 합니다.

3. Canvas 위치는 구조가 아님

nodePositionsByFlow는 App.tsx 안의 UI 상태입니다. 노드를 드래그해도 Flow의 순서나 Branch 관계는 바뀌지 않습니다. 또한 저장되지 않습니다.

4. 검증은 안전망

validateWorkspace는 중복 배치, 고아 Block, 끊긴 Branch, 부모 불일치를 문자열 오류로 돌려줍니다. 새로운 구조 명령은 이 규칙을 깨지 않아야 합니다.

다음 기능을 고민하는 방법

기능 요청을 바로 UI 코드로 옮기지 말고, 아래 순서로 생각해 보세요.

  1. 사용자 행동을 문장으로 쓴다.
    예: “사용자가 앱을 닫아도 Flow와 Block이 복원된다.”
  2. 변하는 데이터와 지켜야 할 규칙을 찾는다.
    이 예시라면 WorkspaceState를 읽고/쓰는 저장 형식, 실패 시 원본 보존, UI 좌표 저장 여부가 쟁점입니다.
  3. 도메인 명령과 테스트를 먼저 정한다.
    UI와 무관하게 저장 전후의 Workspace가 같은 구조인지 테스트할 수 있어야 합니다.
  4. 그 뒤에 App과 Tauri의 책임을 나눈다.
    App은 저장 상태와 버튼을 보여주고, Tauri/Rust는 사용자 디스크 입출력을 담당합니다.

첫 학습 목표: Flow 하나를 만들고 Block 3개와 Branch 1개를 추가한 뒤, 실제로 어떤 함수가 어떤 순서로 호출되는지 위 읽기 순서대로 추적해 보기.
