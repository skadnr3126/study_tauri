FOCUS: IN-MEMORY DATA

추가·삭제할 때 데이터는
메모리에서 어떻게 바뀌나?

현재 앱은 파일이나 DB에 저장하지 않습니다. 화면의 모든 데이터는 React의 workspace 상태 하나에 올라가 있고, 버튼을 누를 때마다 기존 상태를 복제해 새 상태를 만든 뒤 교체합니다.

가장 먼저 잡을 한 가지

Block은 메모의 내용이고, Branch.itemIds는 그 메모가 Flow 안의 어디에 놓였는지를 나타냅니다.

workspace.blocks  → Block ID : 메모 내용
flow.branches.get(branchId).itemIds → 메모의 순서/위치

blocks: blk_a → { id, title, markdown, createdAt, updatedAt }
itemIds: ["blk_a", "blk_b", "blk_c"]

따라서 Block을 추가/삭제할 때는 내용 Map과 위치 배열을 함께 바꿔야 합니다. 하나만 바꾸면 고아 데이터 또는 깨진 참조가 생깁니다.

앱이 시작될 때: 빈 메모리

App.tsx의 첫 상태는 createWorkspace()입니다.

{ blocks: new Map(), flows: new Map(), activeFlowId: undefined, selectedBlockId: undefined }

새 Flow를 만들면 root Branch가 하나 생기지만 아직 Block은 없습니다. 새로고침/앱 종료 뒤에는 이 함수가 다시 실행되므로 모든 데이터가 빈 상태로 돌아갑니다.

공통 규칙: 원본을 바로 바꾸지 않고 새 Workspace를 만든다

1. UI 이벤트: “다음 노드 추가” 버튼 클릭
↓
2. App.tsx의 runCommand가 현재 workspace를 도메인 함수에 전달
↓
3. flow.ts의 cloneWorkspace가 Map·배열을 복제
↓
4. 복제본만 수정하고 { workspace: next }를 반환
↓
5. setWorkspace(next): React가 새 데이터로 화면을 다시 렌더링

이 규칙 덕분에 수정 전과 수정 후 상태가 섞이지 않습니다. 나중에 Undo/Redo를 만들 때도 이 구조가 기반이 됩니다.

Block 추가: 실제로 바뀌는 데이터

createBlockAfter(workspace, flowId, afterBlockId?)가 담당합니다.

추가 전

blocks: blk_a, blk_b
root.itemIds: ["blk_a", "blk_b"]

blk_a 뒤에 추가 후

blocks: blk_a, blk_new, blk_b
root.itemIds: ["blk_a", "blk_new", "blk_b"]
selectedBlockId: "blk_new"
  1. buildLocationIndex로 afterBlockId가 속한 Branch와 배열 인덱스를 찾습니다.
  2. 그 Branch의 itemIds.splice(insertAt, 0, blockId)로 새 ID를 끼워 넣습니다.
  3. blocks.set(blockId, { markdown: "", createdAt, updatedAt })로 빈 내용 Block을 만듭니다.
  4. selectedBlockId를 새 ID로 설정합니다.

정상적인 추가는 위치 배열과 내용 Map을 한 명령 안에서 같이 바꾸므로 고아 Block을 만들지 않습니다.

Branch 추가: Block 하나와 Branch 하나를 동시에 만든다

createBranchWithBlock(workspace, flowId, parentBlockId)는 다음 세 곳을 수정합니다.

1. flow.branches에 새 Branch 추가: { parentBranchId, parentBlockId, itemIds: ["새 Block"] }
2. 부모 Branch의 childBranchIdsByBlockId에 새 Branch ID 연결
3. workspace.blocks에 첫 빈 Block 추가

즉 “갈래 만들기”는 선만 추가하는 기능이 아닙니다. 바로 쓸 수 있는 첫 빈 메모까지 함께 생성합니다.

삭제: 현재 코드에서 반드시 알아야 할 핵심

함수 이름은 deleteBlockSubtree지만, 현재 동작은 선택한 Block 하나와 하위 가지 하나만 지우는 것이 아닙니다.

선택한 Branch의 itemIds가 [A, B, C, D]이고 B를 삭제하면

현재 결과: [A]
삭제 대상: B, C, D 그리고 B/C/D에서 시작한 모든 하위 Branch

즉, B “이후의 같은 줄 흐름”도 함께 삭제됩니다.

근거는 flow.ts의 removedItemIds = branch.itemIds.slice(location.index)입니다. B의 인덱스부터 배열 끝까지 잘라냅니다.

설계 문서의 “해당 Block과 하위 Branch 전체 삭제”를 “선택 Block만 + 그 하위 Branch만 삭제”로 해석했다면, 현재 구현은 요구와 다릅니다. 이것이 추가/삭제 구조에서 가장 먼저 결정하거나 고쳐야 할 문제입니다.

현재 삭제가 실제로 하는 일

  1. 선택 Block의 위치를 찾습니다.
  2. 그 Block부터 해당 Branch의 마지막 Block까지 removedItemIds로 수집합니다.
  3. 각 삭제 Block 아래의 Branch를 재귀 순회해 삭제할 Block ID와 Branch ID를 모읍니다.
  4. 원래 Branch의 itemIds를 선택 Block 앞까지만 남깁니다.
  5. workspace.blocks.delete(id)로 모은 모든 내용 Block을 제거합니다.
  6. flow.branches.delete(id)로 모은 하위 Branch를 제거합니다.
  7. 삭제 뒤 Branch가 비었고 root가 아니면 부모의 연결 목록에서도 그 빈 Branch를 제거합니다.
  8. selectedBlockId를 비웁니다. UI도 편집 중인 Block ID를 비웁니다.

삭제 후에도 남아 있는 Block이 반드시 어떤 Branch에 한 번만 배치되어 있는지 validateWorkspace가 검사합니다.

데이터 관점에서 확인해야 할 문제 목록

질문현재 답결정/개선 방향
삭제는 어디까지인가?선택 Block부터 같은 Branch의 끝 + 모든 하위 Branch“선택 Block과 그 하위 Branch만”이 목표라면 삭제 알고리즘을 바꿔야 합니다.
삭제한 데이터는 복구 가능한가?아니요. 메모리 Map에서 즉시 제거됩니다.Trash 또는 Undo를 추가하기 전에는 되돌릴 수 없습니다.
화면 위치 데이터도 삭제되는가?아니요. nodePositionsByFlow에 오래된 ID가 남습니다.현재 화면에는 영향이 작지만, 저장/대형 Flow 전에는 정리 로직이 필요합니다.
앱을 닫아도 남는가?아니요. 모든 상태가 React 메모리뿐입니다.저장 기능은 Workspace와 별도 UI 위치 상태를 어떤 형식으로 저장할지 결정해야 합니다.
Block이 두 위치에 나타날 수 있는가?명령은 그렇게 만들지 않으며 검증 함수가 중복을 잡습니다.새 이동/복제 기능도 이 불변 조건을 지켜야 합니다.

이제부터 코드를 읽는 가장 좋은 방법

  1. application/src/domain/flow.test.ts 첫 테스트를 열어 A → B → C와 Branch를 만드는 흐름을 따라갑니다.
  2. 각 줄에서 호출하는 createBlockAfter, createBranchWithBlock, deleteBlockSubtree만 flow.ts에서 찾아 읽습니다.
  3. 각 함수마다 “어느 Map이 바뀌나?”, “어느 배열이 바뀌나?”, “참조를 끊거나 연결하는 Map은 무엇인가?” 세 질문에 답합니다.
  4. 마지막으로 validateWorkspace가 그 결과를 왜 정상이라고 판단하는지 확인합니다.

다음으로 해결할 한 가지를 고른다면: 삭제의 정확한 제품 규칙을 먼저 확정하는 것이 좋습니다. 그 결정이 Trash/Undo/저장/캔버스 정리 방식까지 전부 바꿉니다.
